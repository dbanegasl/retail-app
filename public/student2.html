CI/CD y Automatización con GitHub Actions

Curso: Fundamentos de DevOps potenciado con IA

Conceptos Clave

Término¿Qué es?
🏷️ Git TagUna "fotografía inmutable" del código en un momento exacto. No avanza como una rama.
📦 GitHub ReleaseEntregable empaquetado generado a partir de un tag. Puede incluir assets descargables.
⚙️ GitHub ActionsSistema de CI/CD integrado en GitHub. Ejecuta workflows automáticos ante eventos.
📋 Workflow YAMLArchivo de configuración declarativo que define cuándo y cómo correr el pipeline.
🔐 SecretVariable cifrada almacenada de forma segura. Nunca aparece en logs — se reemplaza por ***.
🔀 Pull RequestSolicitud de fusión de una rama a otra. Puede disparar pipelines automáticos.

Versionado Semántico, Tags y Releases

¿Qué significa v1.0.0?

Formato MAYOR.MENOR.PARCHE:

MAYOR Rompe compatibilidad
MENOR Nueva funcionalidad
PARCHE Corrección de bugs 
# Asegúrate de estar en main actualizado
git checkout main
git pull origin main

# Crea el tag local
git tag -a v1.0.0 -m "Release v1.0.0: Estructura base y modo oscuro"

# Sube el tag a GitHub
git push origin v1.0.0
⚠️ Importante: git push origin main NO sube los tags automáticamente. Debes usar git push origin v1.0.0 o git push origin --tags.
Publicar Release en GitHub
  1. Ir al repositorio → Releases
  2. Clic en Draft a new release
  3. Seleccionar tag v1.0.0
  4. Título atractivo y publicar

Primer Pipeline con GitHub Actions

🧠 Analogía del Chef: El .yml es la receta, el job es el plato, los steps son los pasos de cocina, y el runner (ubuntu-latest) es la cocina limpia en la nube.
Estructura
retail-app/
└── .github/
    └── workflows/
        └── ci.yml
Pipeline ci.yml
name: Pipeline de Calidad Retail App

on:
  push:
    branches: [ "main" ]

jobs:
  build-and-test:
    runs-on: ubuntu-latest

    steps:
      - name: Clonar el repositorio
        uses: actions/checkout@v3

      - name: Simular Análisis de Calidad
        run: |
          echo "Iniciando análisis de SonarQube simulado..."
          echo "Revisando HTML y JS..."
          echo "✅ Código limpio y sin code smells."

      - name: Simular Pruebas Unitarias
        run: |
          echo "Ejecutando tests de la carpeta tests/..."
          echo "✅ logic.test.js - PASSED"
          echo "✅ html.test.js - PASSED"
# Commit y push para activar el pipeline
git add .
git commit -m "ci: agrega pipeline básico de github actions"
git push origin main

Verifica en la pestaña Actions de tu repositorio.

Secrets y Triggers

Objetivos
  • Pipeline se activa automáticamente en un Pull Request
  • GitHub protege datos sensibles con ***
  • Configurar triggers y secrets en GitHub Actions
Paso 1 — Rama feature-seguridad
git checkout -b feature-seguridad
Paso 2 — Configurar Secret
  1. Repositorio → Settings
  2. Secrets and variables → Actions
  3. New repository secret: API_KEY_RETAIL = super-secreto-123
Paso 3 — Modificar ci.yml
# Agregar pull_request al trigger
on:
  push:
    branches: [ "main" ]
  pull_request:
    branches: [ "main" ]
      # Nuevo step al final
      - name: Probar Secret
        run: |
          echo "Probando acceso al secreto..."
          echo "El secreto es: ${{ secrets.API_KEY_RETAIL }}"
          echo "✅ Si ves *** arriba, el secreto está protegido."
Paso 4 — Push y Pull Request
git add .
git commit -m "ci: prueba de secretos y trigger de PR"
git push origin feature-seguridad

Luego crea el Pull Request desde GitHub hacia main.
